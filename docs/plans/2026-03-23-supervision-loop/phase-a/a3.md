```text
## Per-Phase Execution (Wave Loop)

LOOP until all phases complete:
  a. Ready phases: depends_on all in completed set
  b. Reconciliation (non-root phases): unchanged from current
  c. Dispatch ready phases with Agent run_in_background: true
     - For each phase: create worktree (step 1), bootstrap deps (step 3),
       extract context (step 4), dispatch (step 5) — capture agent_id
     - Initialize per-phase tracking: prev_output_len=0, prev_head_sha,
       no_progress_count=0, intervention_count=0

  SUPERVISION LOOP (every 60s):
    Bash("sleep 60")
    For each active phase:
      - Read plan.json → task completion counts
      - Check escalation-*.json in phase worktree → surface to user
      - TaskOutput(agent_id, block: false) → health signals
      - git log --format=%H -1 in phase worktree → commit recency
      - Evaluate: healthy / degraded / stuck
      - If stuck: L1 intervention (see below)

    OUTPUT PROGRESS UPDATE:
      "[{elapsed}m] Phase A: 3/5 tasks, healthy | Phase B: 1/4 tasks, healthy"

    PROCESS COMPLETED PHASES (serially, inline):
      When TaskOutput status == completed → run steps 6-18 inline
      (review loop → rebase → create-pr → poll → review-pr → merge)

    If all phases complete → break
```

### Step 3: Add L1 intervention protocol

Add intervention protocol section:

| Attempt | Action |
|---------|--------|
| 1st | TaskStop(agent_id) + re-dispatch phase dispatcher with diagnosis |
| 2nd | AskUserQuestion — user decides: re-dispatch with guidance, or abort phase |

L1 never auto-kills a phase dispatcher without user consent.

### Step 4: Add progress and escalation handling details

**Escalation handling:** On each poll, check `ls escalation-*.json` in each phase worktree. For each file found, read it and surface the issue to the user via output. The orchestrator does not auto-resolve escalations — it presents them for user awareness and continues.

**Progress format example:**
```text
[1m] Phase A: 1/5 tasks, healthy | Phase B: 0/4 tasks, starting
[2m] Phase A: 3/5 tasks, healthy | Phase B: 1/4 tasks, healthy
[3m] Phase A: 5/5 tasks → review | Phase B: 2/4 tasks, degraded (no commits)
```

### Step 5: Add supervision configuration reference

Add a note in the Setup section or wave loop that supervision intervals are configurable via optional plan.json `supervision` object:

```json
{
  "supervision": {
    "orchestrator_poll_seconds": 60,
    "dispatcher_poll_seconds": 30,
    "max_intervention_attempts": 2
  }
}
```

All fields optional with defaults shown. The orchestrator reads these at startup:
- Uses `orchestrator_poll_seconds` for its own L1 polling interval
- Passes `dispatcher_poll_seconds` as `{DISPATCHER_POLL_SECONDS}` and `max_intervention_attempts` as `{MAX_INTERVENTION_ATTEMPTS}` to the phase dispatcher prompt (new template variables added in A2)
