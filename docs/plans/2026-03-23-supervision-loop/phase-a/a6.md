# A6: Run skill-eval for orchestrate

**Avoid:** Don't skip evals even if the changes seem straightforward. Prompt template rewrites can silently change triggering behavior or cause the model to skip critical workflow steps.

## Steps

### Step 1: Run skill-eval for orchestrate (before variant)

Use the skill-eval skill to evaluate the orchestrate skill against scenarios that cover the new supervision behavior. Run against the unmodified skill first (before variant) to establish a baseline.

Scenarios to test:
1. **Async dispatch:** Given a multi-phase plan with independent phases, the orchestrate skill dispatches phases with `run_in_background: true` rather than synchronous `Agent` calls.
2. **Supervision polling:** The orchestrate output includes periodic progress updates with task counts and health status per phase.
3. **Stuck detection:** Given a stuck phase (simulated via scenario description), the orchestrate skill recognizes stuck signals and initiates intervention via TaskStop.
4. **Escalation surfacing:** Given an escalation-*.json file in a phase worktree, the orchestrate skill reads and surfaces the escalation to the user.
5. **L2 sequential tasks:** The phase dispatcher dispatches tasks sequentially (one at a time) even though each uses background dispatch for supervision.

### Step 2: Run skill-eval for orchestrate (after variant)

Run the same scenarios against the modified skill files.

### Step 3: Compare results

Compare before vs after pass rates. After variant should be >= before variant. If regressions are found:
1. Identify which scenario regressed
2. Read the scenario details and the skill file to understand why
3. Fix the skill file to address the regression
4. Re-run the regressed scenario to confirm the fix

### Step 4: Document results

Report pass rates and any regressions found/fixed.
